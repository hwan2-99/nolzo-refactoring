# ⚠️ 문제: Redis 장애 시 휘발성 특성으로 대기열 정보가 유실될 수 있다.
##  ❓1. Redis 장애 시 어떤 문제가 있을까?
- Redis로 현재 멱등성 보장, 선착순 좌석 예매 시 대기열 진입을 구현했지만, Redis는 메모리 기반의 휘발성 저장소이기 때문에, 장애나 재시작 시 대기열에 저장된 정보가 유실될 수 있다.
- 예를 들어, Redis가 장애를 겪거나 서버가 재시작되면, 대기열에 저장된 대기 순서나 예매 요청 상태가 사라져버리게 된다. 이로 인해 대기열 순서가 깨져 예매 흐름이 비효율적이거나 혼잡해질 수 있다.
- 이로 인해 대기 순서 관리나 선착순 예매에서 예기치 못한 오류나 불편이 발생하게 되며, 사용자의 신뢰도에 큰 영향을 미칠 수 있다.

--- 

# 💡해결 방법
## 해결: 대기열 엔티티를 추가하여 DB 테이블로 관리 및 Redis 재시작 시 DB로부터 데이터 복구
```java
@Entity
@Table(
        name = "reservation_queue_entry",
        uniqueConstraints = {
                @UniqueConstraint(
                        name = "uq_queue_event_member",
                        columnNames = {"event_id", "member_id"}
                )
        },
        indexes = {
                @Index(name = "idx_queue_event_status", columnList = "event_id, status"),
                @Index(name = "idx_queue_active_until", columnList = "active_until")
        }
)
public class ReservationQueueEntry {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "event_id", nullable = false)
    private Long eventId;

    @Column(name = "member_id", nullable = false)
    private Long memberId;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 20)
    private QueueStatus status;

    @Column(name = "queued_at")
    private LocalDateTime queuedAt;

    @Column(name = "active_until")
    private LocalDateTime activeUntil;
}
```
```java
@Transactional
    public void rebuildRedisFromDb() {
        LocalDateTime now = LocalDateTime.now();

        List<ReservationQueueEntry> expiredActiveEntries =
                queueEntryRepository.findByStatusAndActiveUntilBefore(QueueStatus.ACTIVE, now);

        for (ReservationQueueEntry entry : expiredActiveEntries) {
            entry.expire();
        }

        List<ReservationQueueEntry> waitingEntries =
                queueEntryRepository.findByStatus(QueueStatus.WAITING);

        for (ReservationQueueEntry entry : waitingEntries) {
            redisTemplate.opsForZSet().add(
                    getQueueKey(entry.getEventId()),
                    String.valueOf(entry.getMemberId()),
                    toScore(entry.getQueuedAt())
            );
            redisTemplate.opsForSet().add(getManagedEventsKey(), String.valueOf(entry.getEventId()));
        }

        List<ReservationQueueEntry> activeEntries =
                queueEntryRepository.findByStatusAndActiveUntilAfter(QueueStatus.ACTIVE, now);

        for (ReservationQueueEntry entry : activeEntries) {
            redisTemplate.opsForValue().set(
                    getEnterKey(entry.getEventId(), entry.getMemberId()),
                    "ENTER",
                    Duration.between(now, entry.getActiveUntil())
            );

            redisTemplate.opsForZSet().add(
                    getActiveKey(entry.getEventId()),
                    String.valueOf(entry.getMemberId()),
                    toScore(entry.getActiveUntil())
            );

            redisTemplate.opsForSet().add(getManagedEventsKey(), String.valueOf(entry.getEventId()));
        }

        log.info("Redis 복구 완료 - waiting={}, active={}", waitingEntries.size(), activeEntries.size());
    }
```

## 2. 기술 선택 이유
### 2-1. Redis 장애 시 안정적인 복구
- Redis 장애나 초기화 시, DB에 저장된 대기열 상태를 바탕으로 Redis를 복구할 수 있다.
- 이를 통해 Redis의 데이터 유실을 방지하고, 대기열 상태를 안정적으로 유지할 수 있다.

### 2-1-1. 애플리케이션 기동 시 Redis 대기열 자동 복구
- 애플리케이션이 기동 완료되는 시점에 DB에 저장된 대기열 상태를 읽어 Redis를 자동으로 복구한다.
- 즉, Redis를 단순 캐시처럼 사용하는 것이 아니라, DB를 복구 기준 데이터로 보고 Redis의 waiting/active 상태를 다시 구성한다.

```java
@EventListener(ApplicationReadyEvent.class)
public void recover() {
    log.info("애플리케이션 기동 완료 - Redis 대기열 복구 시작");
    queueRecoveryUseCase.rebuildRedisFromDb();
    log.info("Redis 대기열 복구 완료");
}
```

- 이 방식의 장점은 서버 재기동이나 배포 이후에도 Redis 대기열을 별도 수작업 없이 다시 복원할 수 있다는 점이다.
- 또한 `rebuildRedisFromDb()` 내부에서는 만료된 `ACTIVE` 상태를 `EXPIRED`로 정리하고 `WAITING` 상태 사용자는 Redis 대기열로 복원하고 아직 유효한 `ACTIVE` 상태 사용자는 TTL과 함께 입장 상태로 복원한다.

```java
@Transactional
public void rebuildRedisFromDb() {
    LocalDateTime now = LocalDateTime.now();
    expireStaleActiveEntries(now);
    List<QueueEntry> waitingEntries = restoreWaitingEntries();
    List<QueueEntry> activeEntries = restoreActiveEntries(now);

    log.info("Redis 복구 완료 - waiting={}, active={}", waitingEntries.size(), activeEntries.size());
}
```

### 2-2. 영속적인 대기열 데이터 관리
- 대기열 상태를 DB에 영속적으로 저장하여 Redis의 장애나 초기화가 발생하더라도 데이터 손실을 방지하고, 서비스의 지속적인 운영을 보장한다.

### 2-3. 대기열 상태의 추적 및 분석 용이
- DB에 대기열 데이터를 저장함으로써, 대기 시간, 순번 등을 추적하고 분석할 수 있어 고객 경험을 개선할 수 있다.
- 상태 변화 기록을 통해 서비스 최적화가 가능하다.

### 2-4. 데이터의 일관성 보장
- 대기열 상태를 DB에 저장함으로써, 데이터의 일관성을 보장한다.
- Redis 장애 시에도 DB를 기준으로 복구하여, 데이터 충돌을 방지하고 불일치 문제를 최소화할 수 있다.

## 3. 단점 및 트레이드 오프
### 3-1. 성능 저하
- 대기열 상태를 DB에 저장하고 Redis에 동기화하는 과정에서 추가적인 성능 비용이 발생할 수 있다.
- DB에 데이터를 저장하고, 그 상태를 Redis로 복구하는 로직이 두 번의 데이터 처리를 요구하므로 성능 저하가 발생할 수 있다.

### 3-2. DB 부하 증가
- 대기열 상태를 DB에 저장하고 관리하면, 대기열 상태 변경 시마다 DB에 대한 쓰기 작업이 발생한다. 이는 DB의 부하를 증가시키고, 성능 저하를 일으킬 수 있다.
- 대기열 상태를 DB에서 관리하는 방식은 DB에 대한 의존성을 높인다. 장애나 문제 발생 시 DB 장애가 서비스에 영향을 줄 가능성이 커진다.

### 3-3. 다중 시스템 장애 복구의 복잡성
- Redis와 DB 간의 장애 복구 로직이 복잡해진다. Redis가 실패하고, DB에서 상태를 복구할 때, 두 시스템 간의 상태 일관성을 유지하는 작업이 필요하다.
- Redis와 DB 간의 동기화 문제가 발생할 수 있다. 예를 들어, DB는 업데이트되었으나 Redis에 반영되지 않았거나, 반대로 Redis에서만 상태가 변경된 경우, 일관성 문제가 발생할 수 있다.

### 3-4. 복잡한 시스템 구조
- DB와 Redis를 동시에 사용하여 대기열 상태를 관리하는 시스템은 구조가 복잡해질 수 있다. 특히 대기열 상태를 두 시스템에 분산하여 처리해야 하므로 기능 구현 및 테스트가 어려울 수 있다.

## 4. 다른 방법 및 비교
### 4.1. Redis Persistence RDB(Redis Database) 사용
#### 4.1.1. 작동 방식
- 스냅샷 방식으로 데이터를 디스크에 저장한다.
- 일정 주기마다 전체 데이터를 메모리에서 디스크로 스냅샷을 찍어 파일로 저장한다.
- 기본적으로 **dump.rdb**라는 파일로 저장된다.
- 저장 주기는 save 명령어로 설정할 수 있으며, 기본적으로 10분마다 한 번 저장되는 설정이 있다.

#### 4.1.2. 장점
- **빠른 데이터 복구**: RDB 파일이 하나로 되어 있기 때문에, 복구가 빠르다.
- **저장 공간 효율성**: AOF보다 저장 공간을 적게 차지할 수 있다.
- **성능**: RDB는 백그라운드에서 비동기적으로 스냅샷을 찍기 때문에 쓰기 성능에 영향을 적게 미친다.

#### 4.1.3. 단점 및 적합성
- **데이터 유실**: 마지막 스냅샷 이후의 데이터는 유실될 수 있다. 즉, 주기적인 스냅샷 간에 발생한 데이터는 복구되지 않으며, 최대 몇 분의 데이터가 유실될 수 있다.
- **정기적인 스냅샷 설정 필요**: 설정이 없으면 RDB가 스냅샷을 생성하지 않기 때문에 정기적인 백업을 위해 주기적인 설정이 필요하다.

- **결론**: RDB는 스냅샷을 기반으로 데이터를 저장하는데, 스냅샷 주기와 주기 사이에 변경된 데이터는 유실될 수 있기 때문에 부적합하다고 판별했다.
  - 예시로 스냅샷 주기가 5분일 경우 Redis에 문제가 생기면 5분간의 변경된 데이터가 유실된다.

### 4.2. Redis Persistence AOF(Append Only File) 사용
#### 4.2.1. 작동 방식
- 모든 쓰기 명령을 로그 형식으로 기록하여 디스크에 저장한다.
- Redis의 모든 쓰기 작업(SET, INCR, DEL 등)이 AOF 파일에 순차적으로 기록된다.
- AOF 파일은 **appendonly.aof**라는 파일로 저장돤다.
- AOF 파일은 fsync 옵션을 통해 디스크에 기록하는 방식을 설정할 수 있다
  - everysec: 매 초마다 데이터를 디스크에 기록
  - always: 모든 명령이 디스크에 기록될 때마다 기록
  - no: 명령을 메모리에서만 처리하고, Redis 종료 시에만 디스크에 기록

#### 4.2.2. 장점
- **데이터 무손실**: AOF는 모든 명령을 기록하므로, 장애 후 데이터를 100% 복구할 수 있다.
- **더 높은 데이터 안전성**: 주기적인 스냅샷과는 달리, 모든 변경 사항이 기록되므로 데이터 유실이 발생하지 않음.
- **복구 시 정확한 복원**: AOF는 정확한 복구를 보장하며, 트랜잭션 순서도 그대로 유지된다.

#### 4.2.3. 단점 및 적합성
- **디스크 성능 의존성**: AOF는 모든 명령을 디스크에 기록하므로 디스크 성능에 의존하며, 디스크 공간을 많이 차지할 수 있다.
- **파일 크기**: AOF 로그가 쌓이면서 파일 크기가 커질 수 있어, 파일 압축이나 재작성이 필요하다.

- **결론**: 로그가 쌓이면서 파일 압축의 추가적인 로직이 필요해 적합하지 않다고 판별했다.
