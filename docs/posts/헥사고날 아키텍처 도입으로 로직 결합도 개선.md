# ⚠️ 문제: `RedisTemplate`에 서비스 레이어가 직접 의존하여 결합도가 높아진다.

- 기존 대기열 로직은 서비스 계층에서 `RedisTemplate`을 직접 사용하고 있었다.
- 이로 인해 서비스가 비즈니스 로직뿐 아니라 Redis 자료구조, key 생성 규칙, TTL 처리 방식까지 함께 알고 있는 구조가 되었다.
- 그 결과, 서비스 레이어가 인프라 구현 세부사항에 강하게 결합되었고 유지보수성과 테스트 용이성이 떨어질 수 있는 상태였다.

<br>

## ❓1. 기존 구조의 문제
### 1.1. 서비스 계층의 책임이 과도하게 집중
- 기존 `QueueService`는 여러 책임을 동시에 가지고 있었다.
  - 대기열 진입 및 입장 가능 여부 판단
  - Redis key 생성 규칙 관리
  - Redis Sorted Set / Set / TTL 처리
  - DB 상태 동기화
  - Redis 복구 로직 처리

- 즉, 하나의 서비스 클래스가 비즈니스 로직과 인프라 로직을 함께 처리하고 있었다.

### 1.2. 변경 영향 범위가 커짐
- Redis key 규칙이 바뀌거나 저장 전략이 달라질 경우 서비스 코드까지 함께 수정해야 한다.
- RedisTemplate 대신 다른 구현 기술을 사용하려고 해도 서비스 코드도 수정해야 했다.

### 1.3. 테스트가 어려움
- 서비스 계층 테스트 시에도 Redis 구현 세부사항을 고려해야 하는 구조였다.
- 인터페이스 없이 직접 결합되어 있기 때문에 단위 테스트에서 대체 구현을 사용하기가 어렵다.

<br>

# 💡해결 방법
## 해결: 헥사고날 아키텍처를 적용하여 서비스와 Redis 의존성 분리

- 서비스 계층이 더 이상 `RedisTemplate`에 직접 의존하지 않도록 구조를 변경했다.
- 애플리케이션 서비스는 대기열 유스케이스만 담당하고, Redis 및 DB 접근은 포트와 어댑터 뒤로 분리했다.

<br>

## 2. 구조 개선 방향
### 2.1. 애플리케이션 서비스는 유스케이스만 담당
- `QueueService`는 이제 다음과 같은 흐름 제어에 집중한다.
  - 기존 입장 상태 유지 여부 판단
  - 즉시 입장 가능 여부 판단
  - 대기열 진입 처리
  - 대기열 처리 및 복구 흐름 제어

- 즉, 서비스는 "무엇을 할지"만 결정하고 "어떻게 저장할지"는 직접 다루지 않는다.

### 2.2. 포트를 통해 외부 시스템 의존성 추상화
- 서비스는 구체 구현이 아니라 포트 인터페이스에 의존하도록 변경했다.
- 예를 들어 다음과 같은 역할을 포트로 분리했다.
  - `QueueStorePort`: Redis 기반 대기열 상태 관리
  - `QueueEntryPort`: DB 기반 대기열 엔티티 조회 및 저장
  - `MemberPort`: 회원 존재 여부 확인

### 2.3. 어댑터가 기술 구현 담당
- Redis 관련 구현은 Redis 어댑터로 이동했다.
- DB 접근은 persistence 어댑터로 이동했다.
- key 생성 규칙도 별도 컴포넌트로 분리했다.

즉, 실제 구조는 다음과 같이 나뉜다.

```text
Application Service
  -> Port Interface
      -> Redis Adapter
      -> Persistence Adapter
```

<br>

## 3. 변경 전과 변경 후 구조
### 3.1. 변경 전

```text
QueueService
 ├─ RedisTemplate 직접 사용
 ├─ Redis key 생성
 ├─ Sorted Set / Set / TTL 처리
 ├─ DB 상태 변경
 └─ Redis 복구 로직
```

- 서비스가 Redis 구현 세부사항을 직접 알고 있다.
- 하나의 클래스에 책임이 집중되어 있다.

### 3.2. 변경 후

```text
QueueService
 ├─ QueueStorePort
 ├─ QueueEntryPort
 └─ MemberPort

RedisQueueStoreAdapter -> RedisTemplate
QueueEntryPersistenceAdapter -> QueueEntryRepository
MemberPersistenceAdapter -> MemberRepository
```

- 서비스는 포트에만 의존한다.
- Redis 구현은 어댑터가 담당한다.
- DB 접근도 별도 어댑터로 분리된다.

<br>

## 4. 패키지 구조 개선
### 4.1. 기존 구조
- queue 관련 로직이 `reservation` 패키지 내부에 포함되어 있다.

### 4.2. 변경 후 구조
- queue 관련 코드를 별도 패키지로 분리했다.

```text
queue
├─ application
├─ application/port/out
├─ adapter/out/redis
├─ adapter/out/persistence
├─ domain
├─ repository
└─ initializer
```

- 이를 통해 `reservation`은 예약 도메인에 집중하고, `queue`는 대기열 도메인에 집중하는 구조가 되었다.
- 결과적으로 기능별 응집도가 높아지고 책임 경계가 명확해졌다.

<br>

## 5. 헥사고날 아키텍처를 사용한 이유
### 5.1. 서비스와 인프라 구현의 결합도 감소
- 서비스 계층이 `RedisTemplate`에 직접 의존하지 않도록 만들기 위해 결정했다.
- 서비스는 Redis API를 알 필요 없이 기능만 요청하면 된다.

### 5.2. 유지보수성 향상
- Redis key 전략, TTL 정책, 자료구조 변경이 필요해져도 변경 범위가 어댑터 내부로 제한된다.
- 비즈니스 정책 변경과 저장 방식 변경이 분리된다.

### 5.3. 테스트 용이성 향상
- 포트 기준으로 mock 또는 fake 구현을 사용한 테스트가 쉬워진다.
- 서비스 계층 테스트에서 Redis 구현을 직접 고려하지 않아도 된다.

### 5.4. 가독성 향상
- 서비스 코드를 읽을 때 Redis 명령어보다 비즈니스 흐름이 먼저 보이게 된다.
- 대기열 입장, 대기, 복구 같은 유스케이스가 더 잘 드러난다.

<br>

## 6. 다른 방법 및 비교
### 6.1. Facade 또는 Helper 클래스로 추상화
- Redis 접근과 key 생성 로직을 별도 Helper 또는 Facade 클래스로 감싸는 관리한다.
- 이 방식은 중복 코드를 줄이고 서비스 메서드의 길이를 줄이는 데 도움이 된다.
- 하지만 서비스가 결국 특정 구현 클래스에 직접 의존하게 되면, 구조적으로는 인프라 의존성이 그대로 남는다.
- 즉, 코드를 감싼 것에 가깝고 의존 방향 자체를 바꾸지는 못한다.

- **결론**: 구현을 일부 정리할 수는 있지만, 서비스와 인프라의 경계를 명확히 나누기에는 부족하다고 판단했다.

### 6.2. 전통적인 Layered Architecture 안에서 역할만 세분화
- 기존의 `service - repository` 구조를 유지한 채 역할만 조금 더 세분화하는 방법
- 익숙하고 구현 비용이 낮다는 장점이 있다.
- 다만 대기열처럼 Redis와 DB를 함께 조율해야 하는 기능에서는 서비스가 여러 저장소 구현 세부사항을 직접 아는 구조가 되기 쉽다.
- 이 경우 비즈니스 로직과 인프라 로직이 다시 한 계층 안에 섞일 가능성이 높다.

- **결론**: 단순 CRUD 중심 기능에는 적합할 수 있지만, Redis와 DB를 함께 다루는 대기열 로직에는 구조적 한계가 있다고 판단했다.

### 6.3. 헥사고날 아키텍처 선택 이유
- 이번 문제의 핵심은 코드를 단순히 분리하는 것이 아니라, 서비스 계층이 `RedisTemplate` 같은 인프라 구현 세부사항을 직접 알지 않도록 만드는 것이다.
- 헥사고날 아키텍처를 적용하면 서비스는 포트 인터페이스에만 의존하고, 실제 Redis/JPA 구현은 어댑터로 분리할 수 있다.
- 이를 통해 결합도를 낮추고, 유지보수성과 테스트 용이성을 함께 확보할 수 있다.

- **결론**: 단순 추상화보다 의존 방향 자체를 개선할 수 있다는 점에서 헥사고날 아키텍처가 더 적합하다고 판단했다.

<br>

## 7. 단점 및 트레이드 오프
### 7.1. 파일 수와 계층 수 증가
- 포트와 어댑터를 분리하면서 클래스 수와 패키지 수가 증가했다.
- 작은 규모 프로젝트에서는 처음에 다소 복잡하게 느껴질 수 있다.

### 7.2. 추상화 비용 발생
- 단순한 기능에도 인터페이스와 구현체를 나누기 때문에 초기 설계 비용이 증가한다.
- 구조적 이점을 얻는 대신 코드 양은 다소 늘어난다.

### 7.3. 과도한 추상화 가능성
- 필요 이상으로 포트를 세분화하면 오히려 이해하기 어려운 구조가 될 수 있다.
- 따라서 실제로 변경 가능성이 높은 경계만 분리하는 것이 중요하다.

<br>

## 8. 정리
- 기존 구조에서는 서비스 계층이 `RedisTemplate`과 직접 결합되어 있어 비즈니스 로직과 인프라 구현이 함께 섞여 있었다.
- 이를 해결하기 위해 헥사고날 아키텍처를 도입하여 서비스는 포트에만 의존하고, Redis 및 DB 접근은 어댑터로 분리했다.
- 또한 queue 관련 책임을 별도 패키지로 분리하여 `reservation` 도메인과의 경계를 명확히 했다.
- 결과적으로 결합도를 낮추고, 유지보수성, 테스트 용이성, 가독성을 높이는 구조로 개선할 수 있었다.
