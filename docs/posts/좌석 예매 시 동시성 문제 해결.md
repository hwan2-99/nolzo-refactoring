# ⚠️ 문제: 동일 좌석에 대한 동시 접근으로 동시성 충돌이 발생

## 1. 동시 티켓 예매 시 어떻게 문제가 발생할까?
 - 모든 사용자가 순차적으로 예매를 진행하지 않는다.
 - 멀티 스레드 환경에서 외래 키 제약 조건이 있는 테이블에서 레코드를 삽입할 때 교착상태가 발생할 수 있다.

```java
@Entity
@Table(
        name = "reservation",
        uniqueConstraints = {
                @UniqueConstraint(name = "uk_reservation_idempotency_key", columnNames = {"idempotency_key"})
        }
)
public class Reservation extends BaseEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "reservation_id")
    private Long id;

    @Enumerated(EnumType.STRING)
    private ReservationStatus status;

    private int totalPrice;

    private String reservationNumber;

    @Column(name = "idempotency_key", updatable = false, length = 64)
    private String idempotencyKey;

    @OneToMany(mappedBy = "reservation", cascade = CascadeType.REMOVE)
    private List<Ticket> tickets = new ArrayList<>();

    @ManyToOne
    @JoinColumn(name = "member_id")
    private Member member;
}
```

## ❓동시성 처리를 하지 않으면 생기는 문제
- 하나의 좌석이 여러 사용자에게 판매될 수 있다.
- 데이터 정합성을 깨뜨리고 서비스 신뢰도를 무너뜨릴 수 있다.
---

# 💡해결 방법

## 해결 : 해결: JPA `@Lock`을 활용한 DB 비관적 락
```java
public interface SeatRepository extends JpaRepository<Seat, Long> {
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT s FROM Seat s WHERE s.id = :id")
    Optional<Seat> findByIdWithPessimisticLock(@Param("id") Long id);
}
```
<br>

## 2. 기술 선택 이유
### 2-1. 간단한 구현 난이도
- 어플리케이션 레이어에 동시성 제어 로직 구현할 필요가 없음
  - 어노테이션 하나만 레포지토리 레이어에 추가함으로써 간단한 락 적용
<br>

### 2-2. 강력한 데이터 일관성 보장
- 데이터 충돌이 빈번한 환경에서 강력한 데이터 일관성을 보장함
  - 데이터 무결성을 즉각적인 잠금(Lock)으로 완벽하게 보호해준다.
- 충돌을 사전에 방지하므로 롤백 및 재시도 비용이 낮다
<br>

## 3. 단점 및 트레이드 오프
### 3-1. 성능 저하
- 락을 획득 동안 다른 트랜잭션이 대기하는 경우, 동시성 높은 환경에서 성능 병목 현상이 발생할 수 있다.
  - 락 장기 유지 또는 여러 트랜잭션의 동일 리소스 접근 시 성능 저하가 두드러짐

### 3.2. 데드락 발생 가능
- 여러 트랜잭션이 서로 다른 순서로 락 획득을 시도 시, 데드락이 발생할 수 있다.
  - 락 획득 순서의 일관성을 유지하고, 락 타임아웃을 적절하게 설정하는 것이 중요

### 3.3. 데이터베이스 의존성
- 사용하는 DB별로 락 동작 방식이 다르다.

## 4. 다른 방법 및 비교
### 4-1. synchronized
```java
@Service
public class ReservationService {
  @Transactional
  public synchronized ReservationResponse create(Long memberId, ReservationRequest request) {
      Member member = memberRepository.getOrThrow(memberId);
      Reservation reservation = new Reservation(ReservationStatus.PENDING, request.calculateTotalPrice(),
                createReservationNumber(), member);

      seatService.updateWithReservation(request.seats());
      return ReservationResponse.from(reservationRepository.save(reservation));
    }
}
```
#### 4.1.1. 작동 방식
- @Transactional이 붙은 메서드가 호출되면, Spring은 프록시 객체를 생성한다.
- 해당 프록시 객체는 원본 객체를 감싸며, 메서드 호출 전 후로 transaction begin, commit을 수행한다.
- 여기서 프록시 객체에서 수행되는 transaction begin, commit 코드는 synchronized의 영향을 받지 않는다.
<br>

#### 4.1.2. 장점
- java에서 한 자원에 synchronized 키워드를 붙이면, 멀티 스레드 환경에서 단일 스레드만이 해당 자원에 접근가능하도록 보장한다.

#### 4.1.3. 단점 및 적합성
- synchronized 키워드는 한 스레드가 메서드 작업을 완료할 때까지 다른 스레드들은 대기해야 한다. 
  - 이로 인해 프로그램의 성능이 저하될 수 있다.
- 하나의 프로세스 안에서만 보장이 되기 때문에, 서버가 여러 대인 분산 환경에서는 데이터의 정합성을 보장할 수 없다.


- **결론**: 예매 사이트는 고트래픽 환경과 분산 확장을 고려해야 하므로, synchronized만으로는 한계가 있으며 데이터 정합성을 보장하기 위해 분산 환경에서도 동작하는 동시성 제어 방식이 필요하다.

<br>

#### 4.2.1. 작동 방식
- 별도의 락 없이 트랜잭션을 진행하고, 커밋 시점에 데이터 변경 여부를 확인
- 일반적으로 버전 필드를 사용하여 충돌을 감지하며, 충돌 발생 시 `OptimisticLockException`을 발생시켜 애플리케이션 레벨에서 재시도 또는 실패 처리가 필요
  - 아래와 같이 재시도 로직을 AOP를 통해 구현해보았다.
```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface Retry {
    //생략
}
```
```java
@Retry
@Transactional
public ReservationResponse create(Long memberId, ReservationRequest request) {
    Member member = memberRepository.getOrThrow(memberId);
    Reservation reservation = new Reservation(ReservationStatus.PENDING, request.calculateTotalPrice(),
            createReservationNumber(), member);

    seatService.updateWithReservation(request.seats());
    return ReservationResponse.from(reservationRepository.save(reservation));
}
```

#### 4.2.2. 장점

- 충돌이 적을 것으로 예상되는 환경에서 락 대기 없이 트랜잭션을 처리하므로, 성능상 이점이 존재

<br>

#### 4.2.3. 단점 및 적합성

- 문제가 생겨 재시도를 할 경우 요청이 들어온 순서대로 처리되는 것이 아니라, 재시도의 타이밍에 따라 결정된다.
- 외부 시스템 연동과 같은 부작용이 있는 트랜잭션이나 트랜잭션 길이가 길어 충돌 확률이 높은 경우에 적합하지 않음
- **결론**: 낙관적 락은 충돌이 적은 일반적인 조회·수정 시나리오에서는 유리하지만, 동일 좌석에 대한 경쟁이 집중되는 예매 시스템에서는 충돌 발생 가능성이 높아 재시도 로직이 빈번해지고, 그에 따라 처리 순서 보장과 사용자 경험 측면에서 적합하지 않다.

### 4.3. Redis 분산 락(Redisson)

#### 4.3.1. 작동 방식
- Redisson Lock은 특정 자원에 대해 Redis에 락 키를 생성하여, 동시에 하나의 요청만 접근할 수 있도록 제어한다.
- 락을 획득한 요청만 임계 구역에 진입할 수 있으며, 다른 요청은 대기하거나 실패 처리된다.
- 작업이 끝나면 락을 해제하고, 일정 시간이 지나면 자동으로 만료되도록 설정할 수 있어 데드락 위험을 줄일 수 있다.

#### 4.3.2. 장점
- 동일 자원에 대한 동시 접근을 제어해 데이터 정합성을 보장할 수 있다.
- Redis를 기반으로 동작해 확장성 있는 동시성 제어가 가능하다.
- 자동 만료 기능을 통해 데드락 위험을 줄일 수 있다.
- waitTime, leaseTime 설정으로 락 동작을 유연하게 조정할 수 있다.

#### 4.2.3. 단점 및 적합성
- 락의 획득, 해제, 만료 시간 관리, 데드락 방지 등 견고한 분산 락을 구현하기 위한 로직이 복잡하다.
- Redis를 단일 락 저장소로 사용하기 때문에, Redis 장애 또는 네트워크 이슈 발생 시 락 처리 안정성이 영향을 받을 수 있다.
- 잘못된 락 범위 설정이나 과도한 점유 시간 설정은 불필요한 직렬화를 유발하여 시스템 처리량을 떨어뜨릴 수 있다.
- 분산 락을 적용할 때 트랜잭션 시작 시점이 적절하지 않으면, 락을 기다리는 동안에도 데이터베이스 커넥션이 점유되어 커넥션 풀이 빠르게 소진될 수 있다.
- 분산 락 자체만으로는 충분하지 않으며, 락 획득 실패 처리, 재시도 정책, 예외 상황 대응 등 애플리케이션 레벨의 보완 로직이 함께 필요하다.

- **결론**: Redis 분산 락은 향후 서버 확장과 고경합 환경을 고려했을 때 유용한 선택지가 될 수 있지만, 락 관리와 예외 처리 등 구현 복잡도가 높고 현재 프로젝트는 단일 서버 환경이기 때문에 이번 단계에서는 적합하지 않다고 판단하였다.
