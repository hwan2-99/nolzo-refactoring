# ⚠️ 문제: 단일 모듈에서는 계층과 도메인 간 의존성이 쉽게 섞일 수 있다.

- 기존의 구조는 하나의 모듈 안에 controller, service, repository, entity, Redis, S3, scheduler와 같은 여러 계층의 코드가 함께 존재했다.
- 패키지로는 역할을 나누고 있었지만, 빌드 단위가 하나였기 때문에 코드상으로는 어디서든 다른 계층의 클래스를 import 할 수 있었다.
- 즉, 규칙을 잘 지키면 문제가 없지만, 구조 자체가 잘못된 의존을 막아주지는 못하는 상태였다.

<br>

## ❓1. 단일 모듈에서 생길 수 있는 문제
### 1.1. API 계층이 서비스 구현체에 직접 의존하기 쉬움
- 단일 모듈에서는 controller가 application port가 아니라 service 구현체를 바로 주입받아도 컴파일상 문제가 없다.
- 예를 들어 예약 API는 외부 요청을 받는 역할만 담당해야 하지만, 단일 모듈에서는 아래처럼 구현체를 직접 알기 쉬운 구조가 된다.

```java
@RestController
@RequiredArgsConstructor
public class ReservationController {

    private final ReservationService reservationService;
    private final SeatService seatService;
}
```

- 이 경우 controller가 "예약 기능을 실행한다"는 계약보다 "ReservationService라는 구현체를 사용한다"는 사실에 의존하게 된다.
- 서비스 내부 구조가 바뀌면 API 계층까지 함께 흔들릴 수 있다.

<br>

### 1.2. 서비스 계층이 인프라 구현을 직접 알게 되기 쉬움
- 단일 모듈에서는 service에서 repository, RedisTemplate, S3 client 같은 기술 구현체를 바로 가져다 써도 구조적으로 막히지 않는다.
- 실제로 대기열 로직에서는 `QueueService`가 Redis 자료구조와 key 규칙을 직접 알고 있으면 다음과 같은 문제가 생긴다.

```java
@Service
@RequiredArgsConstructor
public class QueueService {

    private final RedisTemplate<String, Object> redisTemplate;

    public void validateQueue(Long eventId, Long memberId) {
        redisTemplate.opsForZSet().rank("queue:" + eventId, memberId);
    }
}
```

- 위 구조에서는 대기열 정책과 Redis 구현 세부사항이 같은 클래스 안에 섞인다.
- Redis key 규칙, TTL, 자료구조가 바뀌면 비즈니스 로직까지 같이 수정해야 한다.
- 테스트에서도 Redis 동작을 함께 고려해야 하므로 단위 테스트가 무거워진다.

<br>

### 1.3. 도메인 간 직접 의존이 커질 수 있음
- 예약 도메인은 예약 생성, 취소, 결제 대기 상태 같은 예약 정책에 집중해야 한다.
- 대기열 도메인은 입장 가능 여부, 대기 순번, Redis 복구 같은 대기열 정책에 집중해야 한다.
- 단일 모듈에서는 두 도메인이 같은 모듈에 있기 때문에 예약 서비스가 대기열 구현체를 직접 호출하는 구조가 쉽게 만들어질 수 있다.

```java
@Service
@RequiredArgsConstructor
public class ReservationService {

    private final QueueService queueService;

    public ReservationResponse create(Long memberId, ReservationRequest request, String idemKey) {
        queueService.validateQueue(request.eventId(), memberId);
        // 예약 생성 로직
    }
}
```

- 이 경우 예약 도메인이 대기열 도메인의 구체 서비스 구현을 직접 알게 된다.
- 나중에 대기열을 Redis 기반에서 Kafka, 별도 서버, 외부 시스템 등으로 변경하려고 하면 예약 서비스까지 영향을 받을 수 있다.

<br>

### 1.4. 테스트 범위가 불필요하게 커짐
- 단일 모듈에서는 모든 코드가 같은 빌드 단위에 들어있다.
- service 테스트를 작성하려고 해도 controller, security, redis, jpa, scheduler 설정이 함께 얽히기 쉽다.
- 결과적으로 순수 비즈니스 로직만 검증하고 싶은 테스트도 전체 Spring Context에 가까운 무거운 테스트가 되기 쉽다.

<br>

# 💡해결 방법
## 해결: 멀티모듈을 도입하여 책임과 의존 방향을 빌드 구조로 분리

- 프로젝트를 역할별 모듈로 분리하여 계층 간 의존 방향을 명확히 했다.
- 단순히 패키지를 나눈 것이 아니라, Gradle module 단위로 분리하여 잘못된 의존이 컴파일 단계에서 드러나도록 했다.

```text
nolzo
├─ nolzo-api
├─ nolzo-core
├─ nolzo-queue
├─ nolzo-infra
└─ nolzo-common
```

<br>

## 2. 현재 모듈 구조
### 2.1. 모듈별 책임

| 모듈 | 책임 |
| --- | --- |
| `nolzo-api` | Controller, Scheduler, Security, 애플리케이션 실행 진입점 |
| `nolzo-core` | 예약, 결제, 공연, 좌석, 티켓, 리뷰, 회원 등 핵심 도메인과 유스케이스 |
| `nolzo-queue` | 대기열 도메인, 대기열 유스케이스, 대기열 port |
| `nolzo-infra` | JPA, Redis, S3, ShedLock 등 외부 기술 구현 adapter |
| `nolzo-common` | 공통 응답, 예외, base entity, 공통 유틸 |

<br>

### 2.2. 모듈 의존 방향

```text
nolzo-api
  ├─ depends on nolzo-core
  ├─ depends on nolzo-queue
  ├─ depends on nolzo-infra
  └─ depends on nolzo-common

nolzo-infra
  ├─ depends on nolzo-core
  ├─ depends on nolzo-queue
  └─ depends on nolzo-common

nolzo-core
  └─ depends on nolzo-common

nolzo-queue
  └─ depends on nolzo-common
```

- 핵심은 `nolzo-core`와 `nolzo-queue`가 `nolzo-infra`를 의존하지 않는다는 점이다.
- 따라서 도메인 서비스는 JPA Repository, RedisTemplate, S3 Client 같은 구체 기술을 직접 알 수 없다.
- 외부 기술 구현은 `nolzo-infra`가 담당하고, 서비스는 port `interface`에만 의존한다.

<br>

## 3. 변경 전과 변경 후 비교
### 3.1. 변경 전: 패키지 규칙에 의존하는 구조

```text
single-module
└─ com.noljo.nolzo
   ├─ reservation
   │  ├─ controller
   │  ├─ service
   │  ├─ repository
   │  └─ entity
   ├─ queue
   │  ├─ service
   │  ├─ repository
   │  └─ redis
   └─ global
```

- 패키지는 나뉘어 있지만 같은 모듈 안에 있기 때문에 의존 방향이 강제되지 않는다.
- controller가 service 구현체를 직접 알아도 된다.
- service가 repository 구현체나 RedisTemplate을 직접 알아도 된다.
- 도메인 간 구현체를 직접 호출해도 컴파일상 막히지 않는다.

<br>

### 3.2. 변경 후: 모듈 의존성으로 경계를 강제하는 구조

```text
nolzo-api
└─ controller, scheduler
   -> in port

nolzo-core
└─ domain service
   -> out port

nolzo-queue
└─ queue service
   -> queue out port

nolzo-infra
└─ adapter
   -> repository, RedisTemplate, S3
```

- API 계층은 UseCase port를 통해 기능을 호출한다.
- Core와 Queue 서비스는 out port를 통해 외부 저장소나 기술을 호출한다.
- Infra adapter만 JPA, Redis, S3 같은 기술 구현을 알고 있다.

<br>

## 4. 내 코드 기준 예시
### 4.1. API는 Service 구현체가 아니라 UseCase 계약에 의존

```java
@RestController
@RequiredArgsConstructor
public class ReservationController {
    private final ReservationUseCase reservationUseCase;
    private final SeatUseCase seatUseCase;
}
```

- `ReservationController`는 `ReservationService` 구현체를 직접 주입받지 않는다.
- 대신 `ReservationUseCase`, `SeatUseCase`라는 입력 port에 의존한다.
- 나중에 서비스 구현체가 분리되거나 내부 로직이 바뀌어도 controller는 동일한 UseCase를 바라볼 수 있다.

<br>

### 4.2. Core 서비스는 Repository 구현체가 아니라 PersistencePort에 의존

```java
@Service
@RequiredArgsConstructor
public class ReservationService implements ReservationUseCase {
    private final ReservationPersistencePort reservationPersistencePort;
    private final MemberPersistencePort memberPersistencePort;
    private final EventPersistencePort eventPersistencePort;
    private final PaymentPersistencePort paymentPersistencePort;
}
```

- `ReservationService`는 JPA Repository를 직접 알지 않는다. 
- 예약 생성, 조회, 취소 정책만 담당하고 실제 DB 접근 구현은 port 뒤로 숨긴다.
- `nolzo-core`는 `nolzo-infra`를 의존하지 않기 때문에, core 서비스에서 infra repository를 직접 import하는 구조가 컴파일 단계에서 제한된다.

예를 들어 `ReservationPersistencePort`는 `nolzo-core`에 있는 출력 port이다.

```java
public interface ReservationPersistencePort {
    Optional<Reservation> findById(Long id);
    //...
}
```

그리고 이 port의 실제 구현은 `nolzo-infra`의 adapter가 담당한다.

```java
@Component
@RequiredArgsConstructor
public class ReservationPersistenceAdapter implements ReservationPersistencePort {

    private final ReservationRepository reservationRepository;

    @Override
    public Optional<Reservation> findById(Long id) {
        return reservationRepository.findById(id);
    }

    @Override
    public <S extends Reservation> S save(S reservation) {
        return reservationRepository.save(reservation);
    }

    //...
}
```

- 즉, `ReservationService`는 `ReservationPersistencePort`만 알고 있다.
- `ReservationPersistenceAdapter`가 port를 구현하면서 Spring Data JPA Repository 호출을 대신 수행한다.
- 이 구조 덕분에 DB 접근 방식이 바뀌어도 core 서비스가 아니라 infra adapter 중심으로 변경할 수 있다.

<br>

### 4.3. Redis 구현은 Infra Adapter로 이동

```java
@Component
@RequiredArgsConstructor
public class RedisQueueAdapter implements QueueStorePort {
    private final RedisTemplate<String, Object> redisTemplate;
    private final QueueKeyGenerator keyGenerator;
}
```

- RedisTemplate은 `nolzo-queue`의 서비스가 아니라 `nolzo-infra`의 adapter가 가진다.
- `QueueService`는 Redis Sorted Set, TTL, key 규칙을 직접 다루지 않고 `QueueStorePort`를 통해 필요한 기능만 요청한다.

```java
@Service
@RequiredArgsConstructor
public class QueueService implements QueueUseCase {
    private final QueueStorePort queueStorePort;
    private final QueueEntryPersistencePort queueEntryPort;
    private final QueueMemberLookupPort memberPort;
}
```

- 이 구조 덕분에 대기열 전략과 Redis 구현 세부사항이 분리된다.
- Redis key 전략이 바뀌어도 `QueueService`가 아니라 `adapter`와 `key generator` 중심으로 수정하면 된다.

<br>

### 4.4. 예약 도메인은 대기열 구현체가 아니라 Queue Port에 의존

```java
@Service
@RequiredArgsConstructor
public class ReservationService implements ReservationUseCase {
    private final ReservationQueuePort reservationQueuePort;

    public ReservationResponse create(Long memberId, ReservationRequest request, String idemKey) {
        reservationQueuePort.validateQueue(request.eventId(), memberId);
        // 예약 생성 로직
    }
}
```

- 예약 서비스는 대기열 구현체의 상세 로직을 직접 알지 않는다.
- 예약 생성 전에 대기열 검증이 필요하다는 계약만 알고 있다.
- 대기열 구현이 Redis 기반에서 다른 방식으로 바뀌더라도 예약 서비스의 변경 범위를 줄일 수 있다.

<br>

## 5. 멀티모듈을 하지 않았다면 생길 수 있는 상황
### 5.1. Redis 대기열 정책 변경 시 예약 서비스까지 영향
- 예를 들어 대기열 저장 방식을 Redis Sorted Set에서 Stream 또는 외부 Queue 시스템으로 바꾼다고 가정한다.
- 단일 모듈에서 `ReservationService`나 `QueueService`가 RedisTemplate을 직접 알고 있었다면 Redis API 호출부를 서비스 로직 곳곳에서 수정해야 한다.
- 멀티모듈 구조에서는 Redis 구현이 `nolzo-infra` adapter에 있으므로 변경 범위가 infra 쪽으로 제한된다.

<br>

### 5.2. Repository 구현 변경이 도메인 서비스로 전파
- 예를 들어 예약 조회 성능 개선을 위해 JPA query method를 QueryDSL 또는 MyBatis 기반 구현으로 바꾼다고 가정한다.
- 단일 모듈에서 서비스가 Spring Data JPA Repository에 직접 의존하면, 구현 변경이 service 코드까지 전파될 수 있다.
- 현재 구조에서는 `ReservationService`가 `ReservationPersistencePort`에 의존하므로, 조회 구현 방식 변경은 infra adapter 내부로 숨길 수 있다.

<br>

### 5.3. 테스트가 점점 통합 테스트 중심으로 무거워짐
- 단일 모듈에서는 서비스 테스트를 작성할 때도 실제 Spring 설정, JPA, Redis, Security 설정이 함께 끌려오기 쉽다.
- 그러면 작은 비즈니스 정책 하나를 검증하는 테스트도 느려지고 실패 원인도 복잡해진다.
- 멀티모듈 구조에서는 core와 queue의 service가 port에 의존하므로, port를 mock 또는 fake로 대체하여 비즈니스 로직 중심으로 테스트하기 쉬워진다.

<br>

## 6. 멀티모듈의 장점
### 6.1. 의존 방향이 명확해짐
- core와 queue는 infra를 모른다.
- infra는 core와 queue의 port를 구현한다.
- api는 외부 요청을 받아 usecase를 호출한다.

```text
API -> In Port -> Service -> Out Port -> Infra Adapter -> External System
```

<br>

### 6.2. 도메인 응집도가 높아짐
- 예약은 `nolzo-core`의 reservation 도메인에 집중된다.
- 대기열은 `nolzo-queue`에 분리되어 독립적인 도메인으로 관리된다.
- Redis, JPA, S3 같은 기술 구현은 `nolzo-infra`에 모인다.

<br>

### 6.3. 테스트 전략을 나누기 쉬워짐
- core service는 port를 대체하여 비즈니스 규칙 중심으로 테스트할 수 있다.
- infra adapter는 실제 JPA, Redis, S3 연동 중심으로 테스트할 수 있다.
- api는 controller 요청/응답과 인증 흐름 중심으로 테스트할 수 있다.

<br>

## 7. 다른 방법과 비교
### 7.1. 헥사고날 아키텍처만 적용하는 방법
- 멀티모듈을 사용하지 않고 단일 모듈 안에서 헥사고날 아키텍처 패키지 구조만 적용하는 방법도 있다.
- 예를 들어 하나의 모듈 안에서 `application`, `port`, `adapter`, `domain` 패키지를 나누고, controller는 in port를 호출하고 service는 out port에 의존하도록 만들 수 있다.

```text
single-module
└─ reservation
   ├─ application
   │  ├─ port
   │  │  ├─ in
   │  │  └─ out
   │  └─ service
   ├─ domain
   └─ adapter
      ├─ in
      │  └─ web
      └─ out
         └─ persistence
```

- 이 방식도 서비스가 `Repository`, `RedisTemplate` 같은 구현체에 직접 의존하지 않게 만드는 데 도움이 된다.
- 실제로 대기열 리팩토링에서도 port와 adapter를 분리하면서 `QueueService`가 Redis 구현을 직접 알지 않도록 개선했다.

<br>

### 7.2. 헥사고날 아키텍처만으로는 부족했던 이유
- 단일 모듈에서 헥사고날 아키텍처를 적용하면 구조의 의도는 좋아지지만, 의존 방향을 컴파일 단계에서 강제하기는 어렵다.
- 같은 모듈 안에서는 `service`가 `adapter`나 `repository` 패키지를 import해도 Java 컴파일러가 막지 않는다.
- 즉, 아래와 같은 잘못된 의존이 생겨도 패키지 규칙을 어긴 것일 뿐 빌드 자체는 성공할 수 있다.

```java
@Service
@RequiredArgsConstructor
public class ReservationService {
    private final ReservationRepository reservationRepository;
}
```

- 위 코드는 헥사고날 아키텍처 의도와 맞지 않지만, 단일 모듈에서는 구조적으로 차단되지 않는다.
- 결국 팀원이 규칙을 계속 기억하고 리뷰에서 잡아야 하는 방식이 된다.

<br>

### 7.3. 멀티모듈을 선택한 이유
- 이번 리팩토링의 목표는 단순히 패키지 구조를 정리하는 것이 아니라, 잘못된 의존 방향을 구조적으로 막는 것이었다.
- `nolzo-core`와 `nolzo-queue`가 `nolzo-infra`를 의존하지 않도록 모듈을 분리하면, core나 queue 서비스에서 infra repository, Redis adapter, S3 adapter를 직접 import할 수 없다.
- 즉, 헥사고날 아키텍처의 port/adapter 구조를 유지하면서, 멀티모듈로 의존 방향까지 빌드 구조에서 강제할 수 있다.

```text
헥사고날 아키텍처
  -> port와 adapter로 역할을 분리한다.

멀티모듈
  -> core, queue가 infra를 직접 의존하지 못하게 막는다.

헥사고날 아키텍처 + 멀티모듈
  -> 역할 분리와 의존 방향 강제를 함께 달성한다.
```

- 따라서 이번 프로젝트에서는 헥사고날 아키텍처만 적용하는 것보다, 멀티모듈을 함께 적용하는 것이 더 적절하다고 판단했다.

<br>

## 8. 단점 및 트레이드 오프
### 8.1. 초기 구조가 복잡해짐
- 단일 모듈보다 모듈 수와 gradle 설정이 늘어난다.
- 처음 보는 사람은 어떤 코드가 어느 모듈에 있어야 하는지 학습이 필요하다.

<br>

### 8.2. 클래스 수가 증가함
- port와 adapter를 분리하면서 인터페이스와 구현 클래스가 늘어난다.
- 단순 CRUD 기능까지 과하게 분리하면 오히려 가독성이 떨어질 수 있다.

<br>

### 8.3. 의존성 설계가 필요함
- 어떤 모듈이 어떤 모듈을 알아야 하는지 미리 정해야 한다.
- 특히 도메인 간 협력이 필요한 경우 직접 service를 호출할지, port를 둘지 기준이 필요하다.

<br>

## 9. 정리
- 멀티모듈을 도입한 이유는 단순히 폴더를 깔끔하게 나누기 위해서가 아니다.
- 핵심은 API, 도메인, 대기열, 인프라, 공통 코드의 책임을 나누고 의존 방향을 빌드 구조로 강제하기 위해서이다.
- 단일 모듈에서는 패키지 규칙을 개발자가 계속 의식해야 하지만, 멀티모듈에서는 잘못된 의존이 컴파일 단계에서 드러난다.
- 현재 구조에서는 `nolzo-core`와 `nolzo-queue`가 인프라 구현을 직접 알지 않고, `nolzo-infra`가 port를 구현하는 방향으로 책임이 분리된다.
- 헥사고날 아키텍처만으로도 역할 분리는 가능하지만, 멀티모듈을 함께 적용하여 의존 방향을 더 강하게 제한할 수 있었다.
- 결과적으로 변경 영향 범위를 줄이고, 테스트 용이성을 높이며, 장기적으로 도메인별 유지보수가 쉬운 구조를 만들 수 있다.
