# ⚠️ 문제: 중복 요청으로 인한 예약 처리 문제

---
- 단일 인스턴스 환경에서도 동일한 요청이 여러 번 처리될 가능성이 있다. 
- 특히 예매 시스템처럼 민감한 서비스에서는 이러한 중복 처리로 인해 운영적 문제 및 신뢰도의 문제가 발생할 위험이 있다.
<br>

## 1. 중복 요청 원인

---

### 1.1. 네트워크 중복 요청
- 네트워크 문제: 예매 요청이 타임아웃이나 서버 응답 지연으로 인해 중복 처리될 수 있다.
  - 예: 클라이언트가 요청을 보내고 응답을 기다리는 도중, 타임아웃이 발생해 응답을 받지 못한 경우 동일한 요청을 다시 보내게 되며, 서버는 이를 중복 요청으로 처리할 수 있다.
### 1.2. 중복 요청으로 인한 비즈니스 로직 오류
- 트랜잭션 관리 부재: 요청을 처리하는 중 트랜잭션의 상태나 요청 상태를 제대로 관리하지 않으면 중복 예매가 발생할 수 있다.
  - 예: 예매 요청을 처리하는 중간에 예외 상황이 발생하여, 요청이 중복 처리되거나 예매가 누락되는 경우
### 1.3. 클라이언트 오류
- 사용자 실수: 사용자가 페이지를 새로 고침하거나 '다시 시도' 버튼을 여러 번 눌러서 동일한 요청이 여러 번 전송되는 경우 
  - 예: 웹 애플리케이션에서 버튼 중복 클릭으로 인해 동일한 요청이 서버에 여러 번 전송될 수 있다.

## 2. 시스템의 중복 요청 미처리 시 발생 가능한 문제

---
### 2.1. 중복 예매 및 좌석 충돌
- 동일한 좌석에 대해 여러 번 예매가 발생할 수 있다.
  - 예: 두 사용자가 같은 좌석을 예약할 때, 첫 번째 요청과 두 번째 요청이 동시에 처리되어 중복 예매가 이루어지고, 실제로 두 사람 모두 해당 좌석을 예약한 상태가 될 수 있다.

### 2.2. 금전적 손실 및 이중 결제
- 중복 예매로 인해 이중 결제가 발생하거나, 이미 결제된 좌석이 중복으로 예매되어 고객에게 환불이나 조정이 필요해질 수 있다.
  - 예: 동일한 좌석에 대해 두 번 결제가 이루어지면, 시스템에서 환불 처리가 발생하며, 이로 인해 금전적 손실이 발생한다.

<br>

# 💡해결 방법

---
## 해결: 멱등 키(Idempotency Key)를 Redis에 캐싱하여 중복 처리 방지

- 사용자의 모든 예매 시도 시 **Idempotency Key**를 부여하고, 해당 멱등키를 Redis에서 관리하며 중복 요청을 차단

## 1. 구현 상세

---
### 1.1. @Idempotent

- 멱등성 처리가 필요한 로직 메서드에 적용하는 어노테이션을 커스텀
```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Idempotent {
    // 생략
}
```
### 1.2. IdempotentAspect

- `@Idempotent` 어노테이션이 적용된 메서드가 호출될 경우 메서드보다 먼저 멱등성 로직을 실행
- Spring AOP로 비즈니스 로직과 멱등성 로직을 분리하여, 코드의 응집도를 높임

```java
@Around("@annotation(com.noljo.nolzo.global.aop.idempotent.Idempotent)")
    public Object checkIdempotency(ProceedingJoinPoint joinPoint) throws Throwable {...}

    private Object checkAndSet(ProceedingJoinPoint joinPoint, Idempotent annotation, String idempotentKey)
            throws Throwable {
        try {
            log.info("Redis에 키 설정 시도: {}", idempotentKey);
            Boolean isSet = redisTemplate.opsForValue()
                    .setIfAbsent(idempotentKey, "1", annotation.expiration(), annotation.timeUnit());

            if (Boolean.FALSE.equals(isSet)) {
                Class<? extends RuntimeException> exClass = annotation.exceptionOnDuplicate();
                log.info("멱등성 위반: {}", idempotentKey);
                try {
                    throw exClass.getConstructor(String.class)
                            .newInstance("중복 요청입니다. key=" + idempotentKey);
                } catch (NoSuchMethodException ignore) {
                    throw exClass.getConstructor().newInstance();
                }
            }
            return result;} 
        catch (SeatException e) {...} 
        catch (Exception e) {...}
    }
```
### 1.3. 예약 생성 시 중복 요청 대응

- 예약 엔티티 영속 전 멱등키 검사 및, 중복 요청 차단

```java
@Transactional
@Idempotent(prefix = "reservation:", key = "#idemKey")
public ReservationResponse create(Long memberId, ReservationRequest request, String idemKey) {
    Member member = memberRepository.getOrThrow(memberId);
    int totalPrice = request.calculateTotalPrice();

    Reservation reservation = new Reservation(ReservationStatus.PENDING, totalPrice, createReservationNumber(),
            member, idemKey);

    try {
        reservationRepository.saveAndFlush(reservation);

        seatService.updateWithReservation(request.seats());
        createTicket(request.seats(), reservation);
        return ReservationResponse.from(reservation);
    } catch (DataIntegrityViolationException e) {...}
}
```

## 2. 기술 선택의 이유 및 장점

---

### 2.1. 중복 요청 사전 방지
- 요청 처리의 가장 초기 단계(AOP 인터셉트)에서 중복 요청을 필터링
- 리소스 낭비 방지: 중복 요청을 사전에 차단함으로써 불필요한 DB 접근이나 다른 서비스 호출 등의 자원 소모를 줄일 수 있다.

### 2.2. 핵심 비즈니스 로직 보호 및 시스템 과부하 방지
- 중복 실행을 방지하여 중요한 비즈니스 로직의 불필요한 반복 실행을 막아 시스템의 효율성과 성능을 높일 수 있다.

## 3. 단점 및 트레이드 오프

---

### 3.1. Redis 의존성
- Redis 의존성: 멱등성 처리 로직이 Redis에 의존한다. 즉, Redis가 정상적으로 동작하지 않으면, 멱등성 체크 기능이 작동하지 않을 수 있다.
  - Redis의 장애 감지, 재시도 로직 및 백업 시스템을 마련하여 장애 발생 시 빠르게 대응할 수 있도록 추가적인 보완이 필요하다.

### 3.2. 메모리 사용량
- Redis에서 멱등키를 관리하기 때문에 요청 수가 증가하면 Redis의 메모리 사용량도 증가한다.
- 멱등성 키의 만료 시간을 적절히 설정하여 불필요한 키의 남아있는 시간을 최소화하고, 메모리 누수를 방지하는 것이 중요하다.

## 4. 다른 기술과의 비교 및 적합성

---

### 4.1. DB UNIQUE KEY
- 데이터베이스의 UNIQUE 제약 조건을 사용하여 DB 레벨에서 멱등성을 보장한다. 이를 통해 중복 데이터가 저장되지 않도록 하여, 중복 요청 처리를 DB에서 직접 관리할 수 있다.
- 멱등성 보장이 DB 내에서 이루어지므로, 별도의 외부 시스템 없이도 중복 요청을 방지할 수 있다.
- DB에 부하가 발생할 수 있으며, 성능이 중요한 경우 대용량 트래픽 처리에 제약이 있을 수 있다.

### 4.2. 분산 락(Distributed Lock)
- 분산 락을 사용하여 서버들 간에 자원에 대한 접근을 제어한다. 특정 요청이 처리 중일 때, 다른 서버들이 동일 작업을 실행하지 않도록 막아 멱등성을 보장한다.
- 멀티 인스턴스 환경에서 유용하게 처리할 수 있다.
- **결과**: 해당 프로젝트는 단일 인스턴스 환경이기 때문에 채택하지 않음

## ❓문제: Redis 기반 멱등키 캐싱만으로는 최종적인 멱등성 보장이 어렵다.

--- 

- Redis는 일시적인 캐시 계층이기 때문에 장애, TTL 만료, 네트워크 문제 발생 시 멱등키가 유실될 수 있다.
  - 따라서 DB와 같은 영속 계층에서 최종 상태를 보장하지 않으면 중복 요청이 다시 처리될 가능성이 존재한다.

### 원인 1: Redis 장애

---

- Redis 서버 장애 발생 시 캐싱해 둔 멱등 키 조회/저장 불가


### 원인 2: Redis 데이터 유실

---

- Redis 데이터가 유실되면 기존에 저장된 멱등키가 사라져 이미 처리된 요청을 처음 요청으로 오인할 수 있다.
  - 그 결과 동일 요청이 다시 실행되어 중복 처리(예: 결제, 예약)가 발생할 수 있다.

<br>

# 💡해결 방법

---
## 1. `DB UNIQUE `제약 조건을 통한 최종 멱등성 보장

```java
@Entity
@NoArgsConstructor
@Table(
        uniqueConstraints = {
                @UniqueConstraint(name = "uk_reservation_idempotency_key", columnNames = {"idempotency_key"})
        }
)
public class Reservation extends BaseEntity {...}
```

## 2. 기술 선택의 이유 및 장점

---

### 2.1. DB 레벨에서의 최종 멱등성 보장
- UNIQUE 제약 조건을 통해 동일한 idempotency_key가 중복 저장되는 것을 원천적으로 차단하여, Redis 장애나 캐시 유실 상황에서도 최종적으로 중복 처리를 방지할 수 있다.

### 2.2. 강력한 일관성 보장
- 데이터베이스는 영속 계층이므로 캐시와 달리 장애나 재시작 상황에서도 데이터 일관성을 유지할 수 있다.

### 2.3.운영 안정성 향상
- Redis에서 문제가 생기더라도 DB 제약 조건이 마지막 방어선 역할을 한다.


## 3. 단점 및 트레이드오프

---

### 3.1. DB 부하 증가
- 모든 중복 검증이 최종적으로 데이터베이스까지 도달하므로, 요청량이 많을 경우 DB에 부담이 커질 수 있다.

### 3.2. 예외 처리 필요
- UNIQUE 제약 조건 위반 시 예외가 발생하므로, 이를 중복 요청으로 해석해 적절히 처리하는 애플리케이션 로직이 추가로 필요하다.

## 4. 정리

---

- Redis를 활용해 대기열 처리와 멱등 요청을 빠르게 처리함으로써 성능을 확보하였다.
- 동시에 DB의 UNIQUE 제약 조건을 통해 Redis 장애나 데이터 유실 상황에서도 최종적인 데이터 무결성을 보장하였다.
- 이를 통해 Redis의 성능과 DB의 안정성을 상호 보완하는 구조로 설계하였다.
