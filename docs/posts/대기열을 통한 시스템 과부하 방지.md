# ⚠️ 문제: 좌석 예매 시스템에 과도한 접근이 발생하여 과부하가 발생

## 1. 가정) 단일 인스턴스 환경에서 인기 공연 예매 오픈 시 대규모 트래픽이 유입될 경우, 이를 어떻게 안정적으로 처리할 것인가?
## ❓1.1. 어떤 문제들이 생길 수 있을까?
- 동시 접속 급증으로 인한 서버 과부하
- CPU, 메모리, 커넥션 풀 등의 자원 고갈
- 응답 지연 증가 및 타임아웃 발생
- 일부 요청 실패로 인한 사용자 경험 저하
- 단일 인스턴스 장애 시 전체 서비스 중단 위험
--- 
# 💡해결 방법: 대기열 시스템 도입
## 해결: `Redis` 기반 대기열 시스템 구현

### 1.1. Redis의 **Sorted Set**을 통한 순서 보장 및 대기열 구현
```java
// 예매를 진행할 수 있는 상태인지 검사 및 대기열 진입
public void validateQueue(Long eventId, Long memberId) {
        cleanupExpiredActiveUsers(eventId);
        if (hasEnterKey(eventId, memberId)) {
            syncActiveStateIfNeeded(eventId, memberId);
            return;
        }
        
        long activeCount = getActiveCount(eventId);

        if (activeCount < MAX_ACTIVE_USERS) {
            grantEntrance(eventId, memberId);
            return;
        }

        if (!isAlreadyQueued(eventId, memberId)) {
            addQueue(eventId, memberId);
        } else {
            ensureWaitingState(eventId, memberId);
        }

        Long rank = getQueueRank(eventId, memberId);
        long waitingNumber = rank == null ? 0L : rank + 1L;

        throw new IllegalStateException("현재 예매 대기열에 있습니다. 순번: " + waitingNumber);
    }
```
```java
// 대기열 진입 성공 But 예매 실패 및 대기열 탈출
@Transactional
public void leaveEntrance(Long eventId, Long memberId) {
    redisTemplate.delete(getEnterKey(eventId, memberId));
    redisTemplate.opsForZSet().remove(getActiveKey(eventId), String.valueOf(memberId));

    queueEntryRepository.findByEventIdAndMemberId(eventId, memberId)
            .ifPresent(entry -> {
                if (entry.getStatus() == QueueStatus.ACTIVE) {
                    entry.leave();
                }
            });

    log.info("예매 입장 종료 - eventId={}, memberId={}", eventId, memberId);
}

// 대기열 진입 성공 후 예매 성공 및 대기열 탈출
@Transactional
public void markReserved(Long eventId, Long memberId) {
    queueEntryRepository.findByEventIdAndMemberId(eventId, memberId)
            .ifPresent(ReservationQueueEntry::reserve);

    redisTemplate.delete(getEnterKey(eventId, memberId));
    redisTemplate.opsForZSet().remove(getActiveKey(eventId), String.valueOf(memberId));

    log.info("예약 완료 처리 - eventId={}, memberId={}", eventId, memberId);
}
```

## 2. 기술 선택 이유

### 2-1. 빠른 처리 속도와 높은 처리량
- Redis는 인메모리 기반 데이터 저장소로, 대기열 처리에 필요한 읽기/쓰기 속도가 매우 빠르다.
- 예매와 같이 짧은 시간에 트래픽이 집중되는 상황에서도 안정적으로 요청을 처리할 수 있다.

### 2-2. 동시성 제어에 유리한 구조
- Redis는 단일 스레드 기반으로 동작하여, 동시 요청 상황에서도 데이터 충돌 없이 처리가 가능하다.
- 이를 통해 별도의 락 없이도 대기열 진입 및 active 인원 제어를 안정적으로 수행할 수 있다.
- 예매와 같이 동시 접근이 많은 환경에서 race condition을 줄일 수 있다.

### 2-3. 대기열과 서비스 로직의 분리
- Redis를 활용하여 대기열을 별도로 관리함으로써, 예매 로직과 대기열 로직을 분리할 수 있다.
- 이를 통해 시스템 구조가 명확해지고, 확장 및 유지보수가 용이하다.

## 3. 단점 및 트레이드 오프
### 3-1. Redis 장애 시 데이터 유실 가능성
- Redis는 인메모리 기반이기 때문에 장애나 재시작 시 대기열 데이터가 휘발될 수 있다.
- 특히 티켓팅과 같이 순번이 중요한 서비스에서는 치명적인 문제로 이어질 수 있다.
- 이를 보완하기 위해 DB에 별도 상태 저장 또는 복구 전략이 필요하다.
  - [장애 시 복구 전략 해결](https://github.com/hwan2-99/nolzo-refactoring/blob/develop/docs/posts/%EB%A0%88%EB%94%94%EC%8A%A4%20%EC%9E%A5%EC%95%A0%20%EC%8B%9C%20%EB%B3%B5%EA%B5%AC%20%EC%A0%84%EB%9E%B5.md)

### 3-2. 구현은 빠르지만 운영 복잡도 증가
- 초기 구현은 간단하지만, 장애 대응, 데이터 복구, 상태 정합성 유지 등을 고려하면 운영 난이도가 높아지는 구조다.

## 4. 다른 방법 및 비교
### 4-1. 메시지 큐 기반 (Kafka, RabbitMQ) 대기열
#### 4.1.1. 장점
- 대규모 트래픽 처리에 강하다.
- 이벤트 기반으로 **비동기 처리**가 가능하다.
- 장애 상황에서도 메시지 유실 최소화 할 수 있다.

#### 4.1.2. 단점 및 적합성
- 운영 및 인프라 구성 복잡도가 높다.
- 사용자별 **상태 관리**(입장, 만료, 이탈, 예약 완료)를 위해서는 별도의 상태 저장소와 추가 설계가 필요하다.

- **결론**: 메시지 큐는 높은 처리량과 안정성 측면에서는 장점이 있지만, 본 프로젝트에서 중요한 실시간 순번 조회 및 사용자 상태 기반 대기열 제어 요구사항과는 맞지 않아 채택하지 않았다.
